# Defect RCA: ASW-228 → ASW-239 — Keel Framework Security & Integrity Gaps

**Severity:** P1 (ASW-237, ASW-238) / P2 (ASW-232, ASW-234, ASW-231)
**Environment:** All (dev + CI + production pipeline)
**Affected Version:** ≤3.16.4
**Reported:** 2026-07-23
**CJIS Incident:** Read of input file triggered gate — incident `f64c5cb0092e0376` (NCIC_ID pattern, post-stage, blocked). Input document quoted CJIS category names as audit evidence, not live CJIS data. Gate was technically correct to block. Content referenced by incident hash only; not reproduced here.

---

## Symptoms

- Security scanner (snyk) returning exit 2 (auth/network failure) silently passes pipeline as CLEAN — story proceeds unscanned.
- Audit log terminal entry can be silently tampered; no `self_hash` field protects last written line.
- `config/cjis-patterns.json` contains 5 placeholder/TODO entries — certain identifier categories are declared but not screened. Gate emits no warning and claims full coverage.
- Three scripts referenced in documentation and hooks (`guard-jira-write.cjs`, `guard-approve.cjs`, `tests/weight-budget.test.cjs`) do not exist on disk.
- `agent-output-schema.json` accepts legacy enum values `tdd-red`/`tdd-green` with no deprecation annotation visible to consumers.

---

## Root Cause (per defect)

### ASW-232 — snyk exit-2 fail-open (`scripts/keel-state.cjs:894-897`)

```js
// lines 894-897
const snykFailed = scanners.filter(s => s.name === 'snyk' && s.status === 'ran' && s.exit === 2);
if (snykFailed.length) {
  console.warn('PRESCAN WARNING: snyk returned exit 2 ...');
  // ← pipeline continues to PRESCAN CLEAN
}
```

A `console.warn` then falls through. An invalid `SNYK_TOKEN` silently produces PRESCAN CLEAN. Exit codes > 2 are also not handled.

### ASW-234 — audit hash chain missing `self_hash` (`scripts/keel-state.cjs:137-148`)

```js
function appendAudit(storyId, entry) {
  entry.prev_hash = sha256line(lastLine);
  fs.appendFileSync(p, JSON.stringify(entry) + '\n');
  // ← no self_hash written; last line is unprotected
}
```

`cmdVerify` (line 723) validates `prev_hash` only, so tampering the last entry goes undetected. Genesis sentinel is `'genesis'` string rather than a canonical zero hash.

### ASW-237 — CJIS TODO placeholders (`config/cjis-patterns.json`)

Five `TODO` entries remain (confirmed: `grep -c TODO config/cjis-patterns.json` = 5). Gate loads config, finds no patterns for the placeholder categories, but neither warns nor fails. A story touching those identifier types passes the gate unchecked.

### ASW-238 — Phantom guard scripts

`scripts/guard-jira-write.cjs`, `scripts/guard-approve.cjs`, `tests/weight-budget.test.cjs` — all absent (`ls` exit 2 confirmed). `hooks/hooks.json` does not reference guard scripts (only `keel-classify-gate.cjs` is wired for PreToolUse), so hooks would not fire even if the files existed. Token budget enforcement has no test coverage.

### ASW-231 — Legacy enum lacks deprecation annotation (`agent-output-schema.json:15-30`)

`tdd-red` and `tdd-green` present in enum with no `description` on the `agent` property. Tooling and human readers cannot distinguish active from retired values without reading `keel-state.cjs` source.

## 5-Whys Analysis

**ASW-232**
1. Why did snyk exit 2 pass pipeline? → `console.warn` does not halt execution.
2. Why warn instead of die? → Original fix corrected exit-2 misclassification as DIRTY but did not add a fail-closed path.
3. Why was fail-closed not added? → AC-4 (fail-closed on scan not run) was not verified in the fix PR.
4. Why was AC-4 unverified? → No automated test asserts pipeline exits non-zero on snyk exit 2.
5. Why no such test? → `tests/` has no prescan integration test suite.

**ASW-234**
1. Why is last entry unprotected? → `appendAudit` writes `prev_hash` but not `self_hash`.
2. Why was `self_hash` omitted? → Hash chain implementation covers forward integrity only (can detect tampering of prior entries) but not the terminal entry.
3. Why was terminal entry not considered? → AC-2 (`self_hash` field) was not part of the original implementation scope.
4. Why? → AC spec was added post-implementation.
5. Why? → Audit integrity spec evolved during the security engineer phase without triggering a re-verification cycle.

**ASW-237**
1. Why do TODO placeholders exist? → Categories were declared as a roadmap item; patterns were never implemented.
2. Why does the gate not warn? → `keel-classify-gate.cjs` has no load-time check for empty/placeholder patterns.
3. Why not? → Gate validates pattern syntax but not pattern completeness.
4. Why? → No AC required load-time coverage reporting.
5. Why? → CJIS coverage completeness requirement added post-gate implementation.

**ASW-238**
1. Why do the scripts not exist? → They were documented as requirements but never implemented.
2. Why undiscovered? → Hooks do not reference them, so no runtime error surfaces.
3. Why? → Guard scripts were specced in a separate audit pass after the hook system was finalized.
4. Why not caught in testing? → No test asserts that hooked scripts exist on disk.
5. Why? → Phantom-artifact detection not in CI pipeline.

## Prevention

1. **Fail-closed by default for all security scanners** — any scanner exit ≥ 2 blocks pipeline; opt-out must be explicit per scanner.
2. **Terminal entry self-hash** — every audit append must write `self_hash`; `cmdVerify` must fail on missing `self_hash` for entries that should have it (post-backfill).
3. **Phantom-artifact CI check** — add a lint step: for every script referenced in `hooks.json` and docs, assert the file exists on disk.
4. **CJIS coverage completeness test** — `grep -c TODO config/cjis-patterns.json` must return 0 in CI.
5. **Deprecation annotation discipline** — whenever a schema enum value is retired, the `description` field must be updated in the same commit.
